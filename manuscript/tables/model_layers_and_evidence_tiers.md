## Table 6. Model Layers, Inputs, Outputs, and Evidence Tiers

This table summarizes the main biomathematical and deterministic model layers described in the manuscript. It is intended to clarify which layers are manuscript-central, what they consume and produce, and how strongly each layer is currently supported.

| Model layer | Representative modules | Main inputs | Main outputs | Model form | Current evidence tier | Manuscript-safe interpretation |
| --- | --- | --- | --- | --- | --- | --- |
| HRV analytic core | `app/hrv_core.py` | RR intervals, preprocessing parameters, optional wearable-derived series | Time-domain, frequency-domain, nonlinear, and windowed HRV metrics | Deterministic preprocessing plus standards-informed HRV metric computation | Partial | Strong implementation evidence; external numerical benchmarking remains incomplete. |
| SAFTE-style fatigue and circadian dynamics | `app/fatigue_calculator/safte_model.py`, `frontend/src/lib/safte-model.ts` | Sleep/wake schedule, local time, phase shifts, sleep efficiency, parameter defaults | Percent effectiveness, reservoir state, circadian drive, inertia-modulated fatigue state | Reservoir-based discrete-time biomathematical model with circadian harmonics and inertia | Partial | Strong code-level model specification; client mirror supports architectural consistency, not independent validation. |
| Readiness and scheduling fusion | `app/scheduling_core.py`, `app/scheduling_engine.py` | HRV-derived features, SAFTE state, PVT pathways, hydration/energy modifiers, mission constraints | Readiness-oriented scores, crew-level scheduling context, GO/NO-GO framing | Deterministic rule-based fusion and threshold mapping | Supported | Strongest operational layer for manuscript Results; frame as transparent decision-support logic, not as outcome validation. |
| Task taxonomy and UAS modifiers | `analysis/opi_worked_example.py`, `manuscript/tables/opi_weight_profiles.md`, `manuscript/tables/opi_vigilance_latency_models.md` | Active task category, time-on-task, latency, vehicle count | Task weights, task modifier, vigilance adjustment, latency penalty, multi-vehicle penalty | Linear weighting plus bounded heuristic modifiers | Partial | Safe as framework specification and worked-example instantiation; not yet a separately tested operational pathway. |
| PVT vigilance module | `app/pvt_core.py`, `api/pvt_endpoints.py`, `frontend/src/lib/pvt-scoring.ts` | Reaction times, lapses, false starts, selected PVT variant | Vigilance metrics, lapse counts, operational gate inputs | Deterministic scoring with mirrored Python and TypeScript implementations | Supported | Safe to describe as an implemented and regression-tested vigilance-related input pathway. |
| Sleep analytics module | `app/sleep_core.py`, `api/research_endpoints.py`, `frontend/src/lib/sleep-metrics.ts` | Sleep duration, stage estimates, regularity history, low-SpO2 screening values | Sleep debt, Sleep Regularity Index, stage-balance summaries, screening-only gate outputs | Deterministic wellness-signal scoring and threshold mapping | Supported | Safe to describe as an implemented and regression-tested sleep-context layer, not as PSG-equivalent diagnostics. |
| Longitudinal trajectory-risk layer | `app/trajectory_risk.py` | OPI history, SAFTE effectiveness history, HRV-recovery history | EWMA trajectories, cumulative-strain modifier, longitudinal risk flags | Deterministic smoothing plus thresholded persistence logic | Partial | Safe to describe as implemented longitudinal logic; do not overstate the verification surface because dedicated regression tests are not yet present. |
| API-backed orchestration | `api/main.py`, `api/research_endpoints.py` | Client requests, model parameters, uploaded data, persisted user context | Structured endpoint responses for research and operational routes | Service orchestration over shared model calls | Supported | Safe to describe as the main deployment path for the primary web client. |
| Secondary Streamlit interface layer | `app/research_app.py`, `app/operational_app.py` | Shared backend models and UI state | Legacy or auxiliary research/operational views | Secondary interface over shared models | Supported | Mention as a secondary interface retained in the repository, not as the primary frontend scope of this paper. |
