# Author: Dr Diego Malpica MD

## Table 1. Platform Architecture and Module Families

This table summarizes the main-text architecture and model-delivery layers for the manuscript. It is a system-description table, not a benchmark table.

| Layer or module family | Representative files | Role in the platform | Main manuscript use | Evidence posture |
| --- | --- | --- | --- | --- |
| Physiological analytics core | `app/hrv_core.py`, `docs/Manual.md` | Computes HRV domains, preprocessing pathways, and windowed summaries from RR-interval inputs. | Methods backbone for the computational core. | Implemented and documented; external numerical benchmarking remains incomplete. |
| User context and persistence | `app/user_profile_tab.py`, `app/user_database.py` | Stores longitudinal user characteristics, assessments, and mission-relevant context that can influence downstream interpretation. | Explains how outputs become person-specific rather than context-free. | Supported by code and documented workflow. |
| Fatigue and circadian model layer | `app/fatigue_calculator/safte_model.py`, `frontend/src/lib/safte-model.ts`, `app/frms.py`, `app/frms_v2.py`, `app/fatigue_integration.py` | Converts sleep, circadian, and schedule information into biomathematical fatigue and readiness states, with a canonical Python implementation mirrored in the client. | Core translational layer linking physiology to operational exposure. | Strong code-level model evidence; external benchmarking remains incomplete. |
| Scheduling and decision support | `app/scheduling_core.py`, `app/scheduling_engine.py`, `app/scheduling_tab.py` | Integrates physiological and schedule variables into crew-level scoring, readiness framing, and GO/NO-GO style outputs. | Main operational contribution in Results and Discussion. | Supported by deterministic code paths and representative tests. |
| Space-weather ingestion and alignment | `app/noaa_space.py`, `app/space_weather_impact.py`, `app/space_weather_alignment.py`, `app/space_weather_influence.py` | Ingests, caches, harmonizes, propagates, and aligns environmental context for exploratory coupling with physiological data. | Domain-specific differentiator relative to generic HRV tools. | Supported for implementation and engineering verification; mechanistic physiological interpretation remains limited. |
| Node-first delivery and orchestration | `frontend/`, `api/main.py`, `api/research_endpoints.py` | Exposes one shared model stack through a Next.js frontend in the Node.js ecosystem and a FastAPI orchestration layer. | Supports the primary frontend and deployment claim of the revised manuscript. | Supported by repository architecture, route tree, and API surface. |
| Secondary Streamlit interfaces | `app/research_app.py`, `app/operational_app.py` | Retains auxiliary research and operational views over the same backend models. | Maintains repo accuracy without defining the manuscript's primary frontend scope. | Supported as secondary interface evidence only. |
| Export and audit support | `app/publication_export.py`, `app/export_utils.py`, `logging_config.py` | Produces structured outputs, reproducibility-oriented exports, and audit/logging support. | Supports transparency and reporting sections. | Supported for implementation; best strengthened further by release archiving. |
