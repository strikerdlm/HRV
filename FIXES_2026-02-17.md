# Fixes Applied on 2026-02-17

## Issue 1: Streamlit Cache Warnings in FastAPI Backend

### Problem
When running the FastAPI backend (`uvicorn api.main:app`), you saw warnings like:
```
WARNING streamlit.runtime.caching.cache_data_api: No runtime found, using MemoryCacheStorageManager
```

This happened because:
1. The FastAPI backend imports `app/user_database.py`
2. `user_database.py` used Streamlit decorators (`@st.cache_data`, `@st.cache_resource`)
3. When Streamlit is installed but not running (i.e., no active Streamlit runtime), these decorators try to initialize but fail with warnings
4. The old code only caught `ImportError`, not runtime context issues

### Solution
Modified `app/user_database.py` to check if we're actually running in a Streamlit runtime context before using Streamlit decorators:

```python
def _has_streamlit_runtime() -> bool:
    """Check if we're running in an active Streamlit runtime context."""
    try:
        import streamlit as st
        from streamlit.runtime.scriptrunner import get_script_run_ctx
        return get_script_run_ctx() is not None
    except (ImportError, AttributeError):
        return False

# Use Streamlit caching ONLY if runtime is active
if _has_streamlit_runtime():
    # Use @st.cache_resource and @st.cache_data
else:
    # Use plain Python dict caching
```

### Result
✅ FastAPI backend now starts without Streamlit warnings
✅ Streamlit apps still benefit from caching when running in Streamlit
✅ No performance impact - non-Streamlit contexts use efficient dict-based caching

## Issue 2: Garmin Connect Authentication Not Working

### Problem
Garmin Connect authentication was failing with the message "not authenticating correctly". The issue was in `app/garmin_connect_service.py`:

1. After token-based login, `display_name` might not be populated
2. The code attempted to fetch it using `get_full_name()` but didn't verify success properly
3. If `display_name` remained `None`, subsequent API calls would fail with "None" in the URL path (e.g., `/usersummary/daily/None`)
4. Error handling was insufficient - it only warned but didn't fail the login

### Solution
Enhanced the authentication flow in `app/garmin_connect_service.py`:

**Token-based login flow:**
```python
token_login_succeeded = False
try:
    self.client = Garmin()
    self.client.login(tokenstore_path)
    
    # Always fetch user info to ensure display_name is populated
    full_name = self.client.get_full_name()
    display_name = getattr(self.client, 'display_name', None)
    
    if display_name:
        _LOGGER.info(f"User display_name loaded: {display_name} (full name: {full_name})")
        token_login_succeeded = True
    else:
        raise GarminConnectAuthenticationError(
            "Token login succeeded but display_name not available - tokens may be stale"
        )
    
    if token_login_succeeded:
        return self.client
except (FileNotFoundError, GarthHTTPError, GarminConnectAuthenticationError, 
        GarminConnectConnectionError) as exc:
    _LOGGER.info(f"Token-based login failed ({type(exc).__name__}). Requesting fresh login.")
```

**Fresh login flow:**
```python
# After fresh password login
try:
    full_name = self.client.get_full_name()
    display_name = getattr(self.client, 'display_name', None)
    
    if not display_name:
        raise GarminAuthError(
            "Authentication succeeded but could not retrieve user display name. "
            "This may indicate an issue with the Garmin Connect API or your account."
        )
    
    _LOGGER.info(f"User display_name loaded: {display_name} (full name: {full_name})")
except GarminAuthError:
    raise
except Exception as name_exc:
    raise GarminAuthError(
        "Authentication succeeded but could not retrieve user information. "
        f"Error: {name_exc}"
    ) from name_exc
```

### Key Improvements
1. **Explicit success tracking**: Added `token_login_succeeded` flag to ensure we only return after verified success
2. **Better error messages**: Clear distinction between token login vs fresh login failures
3. **Fail-fast behavior**: Raises exceptions immediately if `display_name` can't be populated
4. **Detailed logging**: Logs both `display_name` and `full_name` for debugging

### Result
✅ Authentication now properly validates that user info is retrieved
✅ Clear error messages if authentication fails at any step
✅ No more "None" in API URLs causing silent failures
✅ Existing tokens in `~/.garminconnect/` are properly validated before use

## Background: Why Streamlit with FastAPI?

**Q: What does Streamlit have to do with the frontend app?**

**A:** This project has **two separate frontends** that share the same Python analysis backend:

1. **Streamlit Frontend** (`app/operational_app.py`, `app/research_app.py`)
   - Python-based UI using Streamlit
   - Great for rapid prototyping and research workflows
   - Uses Streamlit caching for performance

2. **TypeScript/Next.js Frontend** (`frontend/`)
   - Modern web UI with React/TypeScript
   - Uses FastAPI backend (`api/main.py`) as REST API
   - Production-ready, mobile-responsive

**The Problem:**
- The Python modules in `app/` were written for Streamlit
- When FastAPI imports these modules, Streamlit's caching system gets confused
- The fix ensures modules detect which context they're in and adapt accordingly

**Architecture:**
```
┌─────────────────────────────────────┐
│  Streamlit Frontend                 │
│  (app/operational_app.py)          │
│         ↓                           │
│    app/user_database.py ← Uses ST  │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│  Next.js Frontend                   │
│  (frontend/)                        │
│         ↓                           │
│    FastAPI Backend                  │
│    (api/main.py)                    │
│         ↓                           │
│    app/user_database.py ← No ST!   │
└─────────────────────────────────────┘
```

Now both architectures work cleanly without interfering with each other.
